Health AI Query Platform

This project uses Python libraries, Google BigQuery, and Google Vertex. The Python libraries I used are Streamlit and FastAPI.

  1. Streamlit has a bunch of UI components that are ready to use. The input boxes, the buttons, the layouts — everything on the UI is from that library.
  2. FastAPI is the REST backend that takes the form data being sent by the Streamlit front end. It takes that request and makes a call to Vertex.
  3. Vertex is an AI library that lets you call an LLM from the REST API. I have a prompt that tells the AI to generate a query from whatever question the user asked.
  4. Once the LLM creates the SQL query, the REST API makes a call to BigQuery where the data set is stored.
  5. Finally, we take the output of the query and send it back into Vertex and ask it to create a visualization and summary of the data it found.

Healthcare Data Pipeline

This project was an exercise in cleaning data and using data pipelines. It uses Jupyter notebooks, CSV (comma-separated values) files, Mage AI, BigQuery, and Looker Studio.

  1. The dataset was a synthetic Medicare dataset with ICD-9 codes.
  2. I used Jupyter to normalize some of the fields and then output CSV files.
  3. I then used Mage AI, a pipeline tool, to take the CSV files as input. I used a transform stage in the pipeline to convert the data to use ICD-10 codes, which is a more modern standard.
  4. The output of the pipeline was to store the data in BigQuery.
  5. I created some simple Looker dashboards to visualize key aspects of the data.

DICOM Metadata Explorer

This project was to help me understand more details about the DICOM file format as it relates to PACS. It uses existing Python libraries.

  1. The libraries used are pydicom and Streamlit. The UI is built in Streamlit because it has a bunch of existing UI elements already made. The DICOM file format was handled using pydicom because it already understands the DICOM standard and makes accessing that data from Python apps easy.
  2. You load a .dcm file, or choose one of the pre-loaded samples.
  3. pydicom reads all the tags from the selected file and organizes them into clinical groups.
  4. The search field searches across all the tags, and you can click through to see all the different tags.

HL7 v2 to FHIR Integration Testbed

This project was created to help practice troubleshooting HL7 v2 to FHIR message pipelines.

It uses:

  1. A Python script to simulate HL7 v2 messages coming out of an EHR system.
  2. Mirth Connect, which is a healthcare integrations application.
  3. An open-source FHIR R4 REST API called HAPI FHIR.

The setup in Mirth looks like this

  1. A new channel is created (a channel is basically a pipeline). We have to configure the source, the transform, and the destination.
  2. The source of the channel is set to be HL7 v2 messages.
  3. The translation step converts the HL7 input message into JSON (JavaScript Object Notation).
  4. The destination is set to be the FHIR R4 REST API, with JSON as the data type.

How it works

  1. You “send” an HL7 message using the Python script.
  2. In Mirth, you watch the channels for the messages coming into the system.
  3. You can click each individual message, and each individual part of the channel (source, transform, destination) to see the status.
  4. For troubleshooting errors, you can read the log messages for the stage where it failed to understand what the issue is.
